# 后端请求处理流程

这份文档用于说明 YiYang 后端如何接收前端数据、传递请求、处理业务逻辑，以及如何通过 Prisma 和 MySQL 交互。

整体链路可以理解为：

```txt
前端 http/request
  -> gateway HTTP Controller
  -> DTO 参数校验
  -> GatewayJwtGuard 鉴权
  -> GatewayServiceClient 转发 TCP 消息
  -> service-auth / service-care 微服务
  -> 业务 Service
  -> PrismaService
  -> MySQL
```

## 1. 后端如何接收数据

前端请求地址通常是：

```txt
/api/auth/login
/api/customer/beds
/api/customer/check-ins
```

gateway 在 `src/apps/gateway/main.ts` 中设置了全局前缀：

```ts
app.setGlobalPrefix('api')
```

所以：

```ts
@Controller('auth')
@Post('login')
```

最终对应的接口就是：

```txt
POST /api/auth/login
```

登录接口示例：

```ts
@Post('login')
login(@Body() dto: LoginDto) {
  return this.gatewayClient.send(SERVICE_NAMES.auth, AUTH_PATTERNS.login, dto)
}
```

这里的 `@Body()` 会接收前端传来的 JSON body。

如果是路径参数，例如：

```txt
PATCH /api/customer/beds/3
```

后端使用：

```ts
@Param('id') id: string
```

来接收 URL 中的 `id`。

## 2. 数据如何校验

gateway 启动时注册了全局 `ValidationPipe`：

```ts
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,
    transform: true,
    forbidNonWhitelisted: true,
  })
)
```

这表示请求进入 Controller 前，会先根据 DTO 做参数校验和类型转换。

常见 DTO 文件：

```txt
src/modules/auth/dto/login.dto.ts
src/modules/auth/dto/register.dto.ts
src/modules/customer/dto/customer.dto.ts
```

登录 DTO 示例：

```ts
export class LoginDto {
  @IsString()
  @IsNotEmpty()
  mobile!: string

  @IsString()
  @IsNotEmpty()
  password!: string
}
```

含义是：前端必须传 `mobile` 和 `password`，而且它们必须是非空字符串。

如果校验失败，请求不会进入业务逻辑，会直接返回错误。

## 3. gateway 如何传递数据

这个项目不是由 gateway 直接查数据库，而是由 gateway 把请求转发给后面的微服务。

例如登录：

```ts
return this.gatewayClient.send(SERVICE_NAMES.auth, AUTH_PATTERNS.login, dto)
```

其中：

```ts
SERVICE_NAMES.auth
```

表示请求要发给认证服务 `service-auth`。

```ts
AUTH_PATTERNS.login
```

表示消息类型是登录：

```ts
login: { cmd: 'auth.login' }
```

客户、床位、入住、护理等业务会发给 `service-care`：

```ts
SERVICE_NAMES.care
```

对应的消息名定义在：

```txt
src/libs/contracts/auth.contract.ts
src/libs/contracts/care.contract.ts
```

`GatewayServiceClient` 会通过本地注册中心找到服务地址，然后用 NestJS TCP 微服务通信把数据发过去。

## 4. 微服务如何接收数据

### service-auth

`service-auth` 通过 `@MessagePattern()` 接收 gateway 转发过来的消息。

示例：

```ts
@MessagePattern(AUTH_PATTERNS.login)
login(dto: LoginDto) {
  return this.authService.login(dto)
}
```

这表示收到 `{ cmd: 'auth.login' }` 消息时，调用 `AuthService.login()`。

### service-care

客户、床位、入住、退住、护理等业务由 `service-care` 处理。

示例：

```ts
@MessagePattern(CARE_PATTERNS.bedsCreate)
createBed(payload: SaveBedDto) {
  return this.customerService.createBed(payload)
}
```

这表示收到创建床位的消息时，调用 `CustomerService.createBed()`。

所以可以这样理解：

```txt
gateway Controller 负责接收 HTTP 请求
微服务 MessageController 负责接收 TCP 消息
业务 Service 负责真正处理业务
```

## 5. 业务如何处理

业务逻辑主要写在模块的 Service 中。

认证相关：

```txt
src/modules/auth/auth.service.ts
```

客户、床位、入住、退住、膳食、服务关注等：

```txt
src/modules/customer/customer.service.ts
```

护理相关：

```txt
src/modules/nursing/nursing.service.ts
```

登录逻辑大致流程：

```ts
const user = await this.prisma.user.findUnique({
  where: {
    mobile: dto.mobile,
  },
})
```

先根据手机号查用户。

然后校验密码：

```ts
matched = await compare(dto.password, user.passwordHash)
```

密码正确后生成 token：

```ts
const token = await this.jwtService.signAsync({
  sub: user.id,
  mobile: user.mobile,
})
```

最后返回给前端：

```ts
return {
  code: 200,
  message: '登录成功',
  data: {
    token,
    profile,
  },
}
```

客户创建逻辑大致流程：

```ts
await this.ensurePhoneAvailable(payload.phone)
await this.ensureCareLevelExists(payload.careLevelId)

const resident = await this.prisma.resident.create({
  data: {
    fullName: payload.fullName.trim(),
    age: payload.age,
    gender: payload.gender,
    phone: payload.phone.trim(),
  },
})
```

也就是说，Service 通常负责：

- 查重
- 校验关联数据是否存在
- 判断业务状态是否合法
- 清洗和组装数据库字段
- 调用 Prisma 查库或写库
- 返回统一格式的数据

## 6. 数据库如何交互

数据库访问统一通过 `PrismaService`。

文件位置：

```txt
src/prisma/prisma.service.ts
```

`PrismaService` 继承了 Prisma Client：

```ts
export class PrismaService extends PrismaClient
```

业务 Service 中注入后，可以直接调用：

```ts
this.prisma.user.findUnique()
this.prisma.user.create()
this.prisma.resident.findMany()
this.prisma.bed.update()
this.prisma.$transaction()
```

Prisma 的数据模型定义在：

```txt
prisma/schema.prisma
```

数据库连接来自 `.env` 中的：

```env
DATABASE_URL="mysql://root:1111@localhost:3306/yiyang_center"
```

所以数据库交互链路可以理解为：

```txt
CustomerService
  -> this.prisma.bed.create()
  -> Prisma Client
  -> MySQL
```

## 7. 鉴权如何处理

登录、注册接口一般不需要 token。

客户、护理等业务接口会加守卫：

```ts
@UseGuards(GatewayJwtGuard)
```

例如 `CustomerController` 上就使用了：

```ts
@Controller('customer')
@UseGuards(GatewayJwtGuard)
export class CustomerController {}
```

守卫会读取请求头：

```ts
const authorization = request.headers.authorization
```

然后把 token 转发给 `service-auth` 校验：

```ts
AUTH_PATTERNS.validateToken
```

如果 token 有效，就把用户信息挂到请求上：

```ts
request.user = result.data
```

如果 token 缺失、无效或过期，就抛出 `UnauthorizedException`，前端会收到 `401`。

## 8. 错误如何返回给前端

微服务中的错误会先经过：

```txt
src/common/filters/rpc-exceptions.filter.ts
```

它会把异常转换成类似：

```ts
{
  code: 400,
  message: 'xxx',
}
```

gateway 收到微服务异常后，会在：

```txt
src/libs/microservices/client-proxy.util.ts
```

中把它还原成 HTTP 异常。

最后 gateway 的全局异常过滤器：

```txt
src/common/filters/all-exceptions.filter.ts
```

会统一返回给前端。

成功响应通常是：

```json
{
  "code": 200,
  "message": "查询成功",
  "data": {}
}
```

失败响应通常是：

```json
{
  "code": 401,
  "message": "未提供认证凭证",
  "path": "/api/customer/beds",
  "timestamp": "..."
}
```

## 9. 一句话总结

后端整体流程是：

```txt
Controller 接收请求
  -> DTO 校验参数
  -> gateway 转发消息
  -> 微服务 controller 接收消息
  -> service 做业务判断
  -> Prisma 操作 MySQL
  -> 返回 code/message/data 给前端
```
