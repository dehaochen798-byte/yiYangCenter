# Prisma 字段变更流程

这份文档记录 `YiYang_Node` 里最常见的 Prisma 字段变更流程，目标是以后新增一个普通字段时，能在几分钟内稳定完成，不再反复踩坑。

## 适用范围

适用于这类改动：

- 给现有表新增字段
- 修改字段默认值
- 新增索引或唯一约束
- 调整 Prisma schema 后，需要同步数据库结构

不适用于这类改动：

- 只改业务逻辑，不改数据库结构
- 只改 DTO、Service、Controller
- 只改前端展示

## 日常最顺手流程

以“给某个表新增一个普通字段”为例，推荐固定按下面顺序做：

1. 修改 [prisma/schema.prisma](/d:/work/vscode-work/Zzz/YiYang_Node/prisma/schema.prisma)
2. 生成迁移
3. 生成 Prisma Client
4. 修改后端代码
5. 跑构建检查
6. 如果前端要用，再补前端类型和页面

对应命令通常就是：

```powershell
npm run prisma:migrate:dev -- --name add_xxx_field
npm run prisma:generate
npm run build
```

可以把它记成一句话：

`改 schema -> migrate dev -> generate -> 改代码 -> build`

## 标准操作步骤

### 1. 修改 schema

先改：

- [prisma/schema.prisma](/d:/work/vscode-work/Zzz/YiYang_Node/prisma/schema.prisma)

例如：

- 给 `Bed` 加 `remark`
- 给 `Resident` 加 `birthday`

### 2. 生成迁移

在 `YiYang_Node` 目录下执行：

```powershell
npm run prisma:migrate:dev -- --name add_bed_remark
```

迁移命名建议：

- `add_bed_remark`
- `add_resident_birthday`
- `rename_user_mobile_field`
- `add_customer_source`

命名尽量清晰，后面查问题会省很多时间。

### 3. 生成 Prisma Client

虽然 `migrate dev` 通常会顺带处理，但建议明确执行一次：

```powershell
npm run prisma:generate
```

如果你改了 schema，但 TypeScript 还提示字段不存在，第一反应先检查是不是漏了这一步。

### 4. 修改后端代码

常见会涉及这些地方：

- `src/modules/**/dto/*.dto.ts`
- `src/modules/**/**.service.ts`
- 查询返回字段
- 校验逻辑
- `prisma/seed.ts`（如果种子数据要覆盖这个字段）

### 5. 跑构建检查

```powershell
npm run build
```

如果 build 不过，优先检查：

- Prisma Client 是否重新生成
- `seed.ts` 是否受唯一键或新增字段影响
- DTO / Service 是否还在使用旧字段

### 6. 前端联动

如果字段前端要展示或提交，再去补：

- 前端 API 类型
- 表单字段
- 列表展示
- 提交参数

## 什么时候需要迁移

只要是数据库结构变化，原则上都需要迁移。

需要迁移：

- 新增字段
- 删除字段
- 改字段类型
- 改默认值
- 加索引
- 加唯一约束
- 新增表 / 关系

不需要迁移：

- 只改 Service 逻辑
- 只改 DTO
- 只改 Controller
- 只改前端页面

## 本地开发库怎么做

本地开发环境推荐：

```powershell
npm run prisma:migrate:dev -- --name your_change_name
npm run prisma:generate
npm run build
```

如果还要重置并灌种子：

```powershell
npx prisma migrate reset --force
npm run db:seed
```

## 线上/共享库怎么做

线上或共享测试环境不要使用下面这些命令：

- `prisma migrate dev`
- `prisma migrate reset`
- `prisma db push`

线上推荐流程：

1. 本地先生成迁移
2. 提交 `schema.prisma` 和 `prisma/migrations`
3. 在线上执行：

```powershell
npx prisma migrate deploy
```

如果线上已经是历史库、迁移链不干净，不要直接硬跑新迁移，要先做 baseline 方案评估。

## 临时试验怎么做

如果只是本地快速试验，可以用：

```powershell
npx prisma db push
npm run prisma:generate
```

但这只适合临时开发，不适合长期维护流程。

原因是：

- 它不会形成正式 migration 文件
- 其他人环境不容易同步
- 上线风险高

## 常见坑

### 1. 只改 schema，不迁移数据库

表现：

- 代码以为字段存在
- 数据库里其实没有
- 运行时报列不存在

### 2. 改完 schema，忘了 generate

表现：

- TypeScript 仍提示 Prisma 字段不存在

解决：

```powershell
npm run prisma:generate
```

### 3. 改了唯一约束，seed 跟着失效

比如这次 `Bed` 从：

- `@@unique([roomId, bedNo])`

改成：

- `@@unique([roomId, bedNo, deleteVersion])`

那 `prisma/seed.ts` 里所有依赖旧唯一键的 `upsert.where` 都要一起改。

### 4. Windows 下生成 migration.sql 出现 BOM

表现：

- Prisma reset / migrate 执行 SQL 时第一行就报语法错

原因：

- 文件开头多了 UTF-8 BOM，不是 SQL 本身写错

### 5. 软删除只改删除逻辑，不改查询过滤

表现：

- 看起来“删了”
- 实际列表、统计、下拉选择里还继续出现

如果做软删除，通常要同步检查：

- 列表查询
- 关联查询
- 统计查询
- 校验方法
- `seed`

## 这次项目里的经验

这次 `Bed` 软删除踩到的核心问题有这些：

- 本地数据库和历史 migrations 已经漂移
- 直接 `migrate dev` 生成出来的不是纯增量迁移，而是历史补丁混合包
- 复合唯一键修改后，`seed.ts` 需要同步调整
- baseline 迁移文件在 Windows 下要注意 BOM 编码问题

最后本地开发库采用的是：

1. 归档旧 migrations
2. 基于当前 [prisma/schema.prisma](/d:/work/vscode-work/Zzz/YiYang_Node/prisma/schema.prisma) 生成新的 `0_init`
3. `migrate reset`
4. `prisma generate`
5. `db:seed`

## 建议的日常自检清单

每次改字段前后，快速过一遍：

- 我这次是不是改了数据库结构？
- 如果改了，是否已经生成 migration？
- Prisma Client 是否重新生成？
- `seed.ts` 会不会受影响？
- DTO / Service / 查询返回是否需要同步？
- 前端类型和表单是否需要同步？
- `npm run build` 是否通过？
