# A 组个人总结（答辩版）

## 一、个人负责内容

在本项目中，我主要负责 A 组模块的设计、开发与联调，具体包括以下业务内容：

- 客户档案管理
- 员工账号管理
- 房间与床位管理
- 外出登记与归院管理

对应前端入口分别为：

- `/customer/residents`
- `/customer/users`
- `/customer/bed`
- `/customer/outing`

对应后端主要由 `gateway + service-care` 协同完成，请求统一通过 `/api/customer/...` 进入网关，再由网关完成鉴权、权限校验和服务转发。

## 二、主要工作与完成情况

### 1. 完成 A 组核心业务模块的页面与接口建设

我先完成了客户、员工、床位、外出等基础模块的前后端主流程搭建，形成了从页面、接口到数据库的完整闭环。前端侧完成了页面列表展示、弹窗表单、新增编辑、状态展示等功能；后端侧完成了 `Resident`、`User`、`Room`、`Bed`、`Outing` 等核心实体的查询与维护接口，并通过 Prisma 实现数据持久化。

在最新版本中，我进一步将“客户档案”和“员工账号”拆分为两个独立入口，使页面职责更加清晰，也更方便后续按角色做精细化权限控制。

### 2. 补齐表单校验、DTO 校验和异常返回，提升模块稳定性

在开发过程中，我重点完善了前后端双层校验机制。前端对姓名、手机号、年龄、时间等关键字段进行了表单校验，后端通过 DTO 和业务校验进一步保证数据合法性，例如：

- 客户手机号唯一
- 员工手机号唯一
- 房间存在后才能新增床位
- 同一房间内床位编号不能重复
- 外出客户必须处于在住状态
- 同一客户不能存在未归院的重复外出记录

同时，我参与完善了异常兜底与统一错误返回逻辑，尤其修复了重复床位编号场景下直接返回 `500` 的问题，使系统可以明确返回可理解的业务提示，提升了前台人员和管理员的使用体验。

### 3. 完成床位管理的关键业务逻辑设计

床位模块是 A 组中业务约束较多的部分。我在这个模块中重点完成了以下工作：

- 支持房间和床位在同一页面联动管理
- 实现床位新增、修改、查询
- 实现床位删除接口，并采用软删除方案保留历史数据
- 删除后自动刷新房间床位数，保证房间统计数据一致
- 对已入住床位限制删除，避免破坏入住业务数据

这一部分不仅实现了基础 CRUD，还兼顾了状态一致性和历史数据保留，体现了我在业务规则梳理和数据约束设计上的思考。

### 4. 完成外出登记与归院流程闭环

在外出模块中，我完成了“新增外出”和“归院登记”两条主流程。为了保证业务准确性，我在实现时加入了多项约束：

- 只允许对在住客户发起外出登记
- 已存在未归院记录的客户不能重复登记外出
- 归院时需要回写实际归院时间
- 外出与归院过程同步维护外出状态字段

通过这一模块，我将简单的表单登记扩展为完整的状态流转流程，使外出业务具备可追踪、可校验、可回溯的特点。

### 5. 完成员工账号管理与密码重置能力

在员工账号模块中，我完成了员工信息的维护逻辑，并新增了密码重置能力。系统支持管理员对员工账号进行新增、编辑和密码重置，新建账号默认密码为 `123456`，后端会进行哈希处理后再入库，兼顾了业务便利性和基本安全性。

这一部分增强了系统的人事基础数据维护能力，也为后续 RBAC 权限体系落地提供了账号基础。

### 6. 参与模块微服务化改造，明确请求流转链路

除了具体业务开发外，我还参与了 A 组相关接口在新架构下的适配工作。A 组请求不再直接由单体后端处理，而是统一进入 `gateway`，再由网关通过服务发现与 TCP 通信转发到 `service-care`。这使系统具备了更清晰的边界，也提升了后续模块扩展与治理能力。

通过这部分工作，我对“前端页面 -> API -> gateway -> 微服务 -> 数据库”的完整链路有了更深入的理解，也提升了我对分层架构和服务解耦的认识。

### 7. 参与消息中间件接入，实现关键操作审计异步化

在 A 组模块中，我还参与了消息中间件的接入，将部分关键操作改造为“主流程完成后异步记录审计日志”的模式。当前已覆盖的典型场景包括：

- 床位删除
- 外出登记
- 外出归院

系统在业务成功后发布领域事件，通过 Redis Stream 交由消息消费者异步写入 `AuditLog`。这样既减少了主流程阻塞，也为后续扩展通知、预警、统计分析等功能预留了接口。

### 8. 完成 RBAC 权限接入，增强系统可用性与安全性

在项目后期，我参与完成了五角色 RBAC 权限接入。A 组模块按照不同角色进行了菜单可见性和接口权限控制：

- 客户档案：全部角色可见，但护理人员仅可查看自己负责的老人，新增编辑仅管理员可操作
- 员工账号：仅管理员可见并可维护
- 床位与外出：仅管理员和前台人员可见并可操作

这一改动使系统从“功能可用”进一步提升到“权限可控”，更加贴近真实养老机构的岗位职责划分。
